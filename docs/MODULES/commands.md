# Commands Module

## Responsibility

`src/commands.rs` currently owns command dispatch, most one-shot command handlers, stable JSON report builders, session-facing reports, support diagnostics, Git-identity reporting, and local workflow formatting. `src/commands/agent.rs` owns `/agent` list/show/spawn handling, sub-agent task selection, agent inspect JSON, sub-agent task projection, and agent next actions. `src/commands/approval.rs` owns `/approval` list/approve/deny/clear handling, approval request text/JSON formatting, action next actions, and approval output writes. `src/commands/btw.rs` owns `/btw` ask/list/answer/clear handling, side-question text/JSON formatting, answer option parsing, action next actions, and by-the-way output writes. `src/commands/completion.rs` owns shell completion command catalogs, script formatting, install/status reports, and completion output writes. `src/commands/config.rs` owns `/config` show/sources/validate/get/set handling, config inspect JSON, config source reports, project config updates, and config validation helpers reused by support commands. `src/commands/credentials.rs` owns `/credentials` status/template/import/set/remove handling, credential file reads and writes, redacted credential status JSON, local setup next actions, and hidden/stdin API key input. `src/commands/context.rs` owns `/context` workspace context collection and text formatting. `src/commands/diagnose.rs` owns `/diagnose` and `/support` option parsing, diagnostics report JSON, redacted support-bundle generation and artifacts, issue templates, and diagnose next actions, while delegating workspace health to the `/doctor` handler and session diagnosis to the `/session` handler. `src/commands/doctor.rs` owns `/doctor` and `/init` handling: doctor option parsing, workspace/shell/provider/test/Git-identity health checks, shell command path resolution, provider readiness reporting and online provider probing, doctor fix application, and doctor report text/JSON formatting; `handle_doctor` is re-exported so `/diagnose` can reuse it. `src/commands/env.rs` owns `/env` check/plan/setup/install/test handling, environment option parsing, environment check/plan/setup/test JSON, environment next actions, and slash-to-deepcli command mapping; the verify/handoff helpers reuse `environment_checks_json`/`environment_status`/`validate_env_target`/`first_line` via `super::`, while the shared `environment_next_actions`/`dedup_preserve_order` helpers stay in `src/commands.rs`. `src/commands/fork.rs` owns `/fork` source selection, session payload cloning, terminal launch handling, dry-run/error/verification reports, fork JSON schema, and fork next actions. `src/commands/git.rs` owns `/git` read/write action dispatch, Git option parsing, dry-run Git action reports, Git inspect JSON, Git output writes, and Git next actions. `src/commands/goal.rs` owns `/goal` show/start/clear/status/gate handling, default goal contract creation and guard-plan generation, goal readiness collection, goal session selection reused by `/round` goal-status summaries, and goal text/JSON formatting. `src/commands/help.rs` owns command help topics, help text formatting, topic alias normalization, and help summaries. `src/commands/logs.rs` owns `/logs` log discovery, safe tailing, redaction, text/JSON reports, and logs output writes. `src/commands/model.rs` owns `/model` show/list/set handling, model inspection JSON, provider runtime display, and project model config updates. `src/commands/opportunities.rs` owns `/opportunities` option and filter parsing, scorecard product-opportunity filtering and next actions, and opportunity text/JSON formatting; like `/recipes` it reuses `build_round_report` and scorecard projection helpers from `src/commands.rs` via `super::`. `src/commands/parser.rs` owns slash command parsing, public slash command variants, aliases, and argument normalization. `src/commands/permissions.rs` owns `/permissions` show/set-mode handling, permission policy JSON, and project permission mode updates. `src/commands/plan.rs` owns `/plan` show/draft handling, clarification question generation, requirements draft Markdown/JSON formatting, side-question queue writes, and plan output writes. `src/commands/preflight.rs` owns `/preflight` check planning, execution summaries, diagnostics, and release-check JSON. `src/commands/privacy.rs` owns `/privacy` Git history/privacy scanning, sensitive path and token detection, blocked-term suppression, redacted privacy text/JSON reports, and privacy scan output writes. `src/commands/productloop.rs` owns the product-loop core trio — `/scorecard`, `/round`, and `/benchmark` — including the scorecard report builder and opportunities, the round report builder and goal/benchmark gates, the full benchmark suite (run/record/status/summary/trends/compare/baselines/show/cleanup), the SOTA baseline next actions and benchmark-baseline helpers, and the stable `deepcli.scorecard.v1`/`deepcli.round.v1`/`deepcli.benchmark.*` JSON schemas. It re-exports `build_round_report`, `RoundReport`, `ScorecardOpportunity`, the `scorecard_*` projection helpers, `sota_baseline_next_actions`, `local_action_checklist`, and the `DEFAULT_ROUND_SCORE_THRESHOLD`/`DEFAULT_BENCHMARK_*` constants so `/recipes`, `/opportunities`, and other modules can keep consuming them via `super::`. `src/commands/prompt.rs` owns `/prompt` list/get/render/save/delete handling, prompt inspect/render JSON, prompt render argument projection, prompt output writes, and prompt next actions. `src/commands/quickstart.rs` owns `/quickstart` onboarding checks, readiness JSON, and startup next actions. `src/commands/recipes.rs` owns `/recipes` topic normalization, the recipe catalog, SOTA product-loop recipe state, and recipe text/JSON formatting; it reuses `build_round_report`, the SOTA baseline next actions, and scorecard-opportunity projection helpers from `src/commands.rs` via `super::` until the product-loop cluster is extracted. `src/commands/registry.rs` owns command group metadata, help summary types, and running-safe classification. `src/commands/resume.rs` owns `/resume` preview/candidates handling, resumable session filtering, resume JSON schemas, output writes, and resume next actions. `src/commands/response.rs` owns shared command output helpers, workspace-contained `--output` writes, and structured command exits. `src/commands/selftest.rs` owns `/selftest` readiness checks, local install diagnostics, and selftest JSON. `src/commands/session.rs` owns `/session` subcommand dispatch (list/history/next/diagnose/search/rename/export/prune-empty/tools/trace/restore-backup and the approval/side-question queues), the running-safe `/session` handler, restore-backup preview/apply, resumable-session selection and de-noising, session activity/inspection/diagnosis JSON, and session next-action projection. It re-exports the session selection/projection helpers (`session_metadata_json`, `resolve_session_for_*`, `sessions_with_resumable_context`, `short_id`, `SessionFallbackKind`, etc.) so `/resume`, `/fork`, `/approval`, `/btw`, and the verify/handoff helpers can keep consuming them via `super::`. `src/commands/skill.rs` owns `/skill` list/generate/run handling, project skill loading, skill inspect/run JSON, skill output writes, and skill next actions. `src/commands/status.rs` owns `/status` workspace/session status reports, status JSON formatting, session next-action projection, and status output writes. `src/commands/terminal.rs` owns `/terminal` dry-run/open handling, terminal app inference and validation, terminal command formatting, workspace open next actions, and terminal JSON/text reports shared by fork helpers. `src/commands/test.rs` owns `/test` discover/run handling, test command discovery/run JSON, test command projection reused by environment reports, test read/run argument parsing, and test next actions. `src/commands/timeout.rs` owns `/timeout` show/set/reset handling, timeout JSON, and provider turn timeout config updates. `src/commands/trace.rs` owns `/trace` session audit selection, fallback behavior, audit trace formatting, redacted trace JSON, and trace output writes. `src/commands/usage.rs` owns `/usage` session usage reports, audit usage summaries, diagnostics, text/JSON formatting, and usage output writes. `src/commands/version.rs` owns `/version` text and JSON reports. `src/commands/web.rs` owns `/web` search argument normalization and web search tool dispatch.

## Boundaries

- Commands may normalize aliases and validate arguments, but durable behavior should move toward the relevant domain module.
- Command handlers should not directly mutate session files except through `SessionStore`.
- Command handlers should not execute shell, Git, filesystem write, or network actions outside the tool and permission layers.
- New public commands must update `docs/COMMANDS.md` and focused command contract tests.
- Command group metadata exposed through `CommandHelpSummary` must stay synchronized with `docs/COMMANDS.md`.
- Legacy aliases should remain thin wrappers over canonical commands.

## Documentation Sync

Update `docs/COMMANDS.md` for command names, groups, owners, aliases, running-safe implications, stable schema ownership, or compatibility changes. Update this file if command ownership moves out of `src/commands.rs`.
