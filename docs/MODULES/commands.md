# 命令模块

## 职责

`src/commands.rs` 目前负责 `CommandRouter` 分发、跨模块 re-export 和外置测试模块声明；命令处理器主体位于 `src/commands/*.rs`，大型命令契约测试已迁移到 `src/commands/tests.rs`。`src/commands/shared.rs` 负责无状态共享 helper，包括通用参数读取、正整数解析、命令 输出契约。`src/commands/benchmark_dispatch.rs` 负责 benchmark dispatch、scorecard-compatible benchmark args、benchmark gate dispatch 和 `/benchmark` 子命令路由；它只做 CLI 分发并委派到 benchmark run/status/history/artifact/preset/baseline owner 或 scorecard handler。`src/commands/round_benchmark_gates.rs` 负责 round benchmark gate projection、benchmark trend gate、round benchmark status projection 和 freshness suffix；`/scorecard`、`/round` 与 benchmark suite 输出通过该 owner 复用 benchmark gate/status 展示。`src/commands/round_goal_status.rs` 负责 round goal status、goal readiness projection、goalStatus JSON 和 goal_readiness gate support；`/round` 通过该 owner 复用 goal readiness 会话选择和 projection。`src/commands/round_report.rs` 负责 round report builder、round text/JSON output、round summary JSON、round benchmark suite wrapper、`RoundReport`、`RoundTextInput` 与稳定的 `deepcli.round.v1` 输出拼装；`/round`、`/recipes` 与 `/opportunities` 通过该 owner 复用同一套 round 报告契约。`src/commands/benchmark_artifacts.rs` 负责 benchmark list、benchmark show、benchmark cleanup、artifact projection、artifact 读取排序、`BENCHMARK_ARTIFACT_SCHEMA`、`BenchmarkArtifact` 以及 artifact summary/detail JSON/text projection；`/benchmark list|show|cleanup`、benchmark run/record、benchmark status/history/baseline owner 通过该模块复用同一套 artifact 读取和投影。`src/commands/benchmark_presets.rs` 负责 benchmark presets、preset catalog、required evidence presets、default suite presets、`BenchmarkPreset`、`BENCHMARK_PRESETS` 与 preset resolver；`/benchmark presets`、benchmark run-suite、status 与 baseline owner 通过该模块复用同一套 preset catalog。`src/commands/benchmark_runs.rs` 负责 benchmark run、benchmark record、benchmark run-suite 的执行入口、execution artifact 生成、shell timeout、suite schema、artifact path slug 与 run-suite JSON/text projection；`/round --run-benchmark` 与 `/benchmark run|record|run-suite` 通过该 owner 复用同一套执行 artifact 逻辑。`src/commands/benchmark_baselines.rs` 负责 benchmark baseline-template、baseline inventory、compare-ready baseline 判断、baseline JSON/text projection、`sota_baseline_next_actions` 和 baseline 文件加载；`/round`、`/scorecard`、`/recipes`、`/opportunities` 与 benchmark compare/trends 通过该 owner 复用同一套 SOTA baseline 覆盖状态、`format_benchmark_status_json`、`format_benchmark_status_text`、freshness JSON/summary 和 benchmark artifact preset 匹配 helper；`/round`、`/scorecard`、`/benchmark status` 与 benchmark suite 输出通过该 owner 复用同一套状态判断。`src/commands/benchmark_history.rs` 负责 benchmark summary、benchmark trends、benchmark compare 的历史聚合、趋势计算、baseline comparison、JSON/text projection 和 trend gate 状态判断；`/round`、`/scorecard` 与 `/benchmark summary|trends|compare` 通过该 owner 复用同一套 benchmark history projection。`src/commands/agent.rs` 负责 `/agent` 的 list/show/spawn 处理、子代理任务选择、agent inspect JSON、子代理任务投影以及 agent ## 边界

- 命令可以归一化别名并校验参数，但持久化行为应当向相关领域模块迁移。
- 命令处理器不应直接修改会话文件，只能通过 `SessionStore` 进行。
- 命令处理器不应在工具层与权限层之外执行 shell、Git、文件系统写入或网络操作。
- 新增的公开命令必须更新 `docs/COMMANDS.md` 以及针对性的命令契约测试。
- 新增公开 help topic 必须先登记 `CommandMetadata`；`CommandHelpSummary` 的命令分组和 running-safe 标记必须来自该显式 registry，并与 `docs/COMMANDS.md` 保持同步。
- 旧版别名应保持为规范命令之上的薄封装，并在 `docs/COMMANDS.md` 的删除/降级审计中记录保留、降级或删除决策。

## 文档同步

当命令名称、分组、归属、别名、running-safe 影响、稳定 schema 归属、兼容性或删除/降级决策发生变化时，更新 `docs/COMMANDS.md`。如果命令归属从 `src/commands.rs` 中迁移出去，则更新本文件。
